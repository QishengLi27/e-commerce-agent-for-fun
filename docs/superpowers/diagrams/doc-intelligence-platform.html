🔬 Document Parsing

4-tier architecture from OCR to multimodal LLM. Handles native PDFs, scanned images, complex layouts, tables, and signatures. ColPali for vision-based retrieval — no OCR needed for complex pages.

🤖 Agentic Workflow

LangGraph explicit DAG, not ReAct. Two modes: interactive (SSE streaming, per-user checkpointing) and batch (Prefect orchestration with retry and progress tracking).

👥 Multi-Agent

Supervisor + 5 specialized sub-agents (Parsing, Extraction, Classification, QA, Review). LangGraph Command-based handoff with shared PostgreSQL checkpoint state.

🧠 RAG + Knowledge Graph

Graph-first retrieval for structured queries. Vector fallback for semantic search. Neo4j for document lineage + entity relationships. LlamaIndex for ingestion.

⚙️ AI Harness

Evaluation (RAGAS + custom) → Versioning (LangFuse) → Gradual Rollout (Unleash) → Cost Control (LiteLLM) → Stability (circuit breakers). Every AI change evaluated before production.

🧩 Low-Code Delivery

Two layers: visual workflow builder (Dify/LangFlow) for simple RAG apps, and domain-specific DSL (CEL expressions) for complex extraction/classification rules. Platform provides guardrails.

Why ColPali is a Paradigm Shift

Traditional Pipeline

PDF → OCR → Plain Text → Chunking → Text Embedding → Search. Losses at every step: OCR errors, chunk boundaries cut tables in half, visual cues (bold, position, signatures) are lost.

ColPali Pipeline

PDF Page Screenshot → Vision Encoder → Multi-Vector Embedding → Search. No OCR. No chunking. The model sees the page as a human would. "The signature is in the bottom-right corner" — ColPali can retrieve this.

Task Decomposition — Multi-Step Reasoning

Example: "Extract all financial obligations from this 45-page contract and flag any clauses with penalties."
S1
Task Decomposition (LLM CoT): Break into sub-tasks → (1) Identify financial sections, (2) Extract amounts/dates/parties, (3) Classify obligation type, (4) Flag penalty clauses, (5) Cross-reference amounts with summary table, (6) Generate structured output
S2
Parallel Dispatch: Sub-tasks 1&2 run in parallel on different sections. Penalty flagging (sub-task 4) runs independently. LangGraph Send() API fans out to parallel nodes.
S3
Tool Calling: Extraction Agent calls extract_financial_entities(). Penalty Agent calls flag_clause_pattern(). Cross-reference Agent calls validate_against_summary().
S4
Merge + Verify: Cross-check — do extracted amounts sum to the total in the summary table? If discrepancy > tolerance → flag for human review.

Two Operational Modes

Interactive Mode

User-facing chat/query. Real-time SSE streaming. LangGraph checkpoint per-user session. Context compression (sliding window + token budget). Circuit breaker + retry per external call.

Batch Mode

Async document processing. Prefect workflow orchestrator. Schedule: nightly batch, webhook-triggered. Progress tracking + partial failure handling + retry. Output → structured DB.

Agent Specialization Details

📄 Parsing Agent

Tools: parse_document(), detect_layout(), ocr_escalate()

Strategy: Layout complexity < 0.3 → Tier 2. Complexity > 0.6 + needs reasoning → Tier 4 (Multimodal LLM). Otherwise → Tier 3 (ColPali).

🔍 Extraction Agent

Tools: extract_entities_ner(), extract_key_value(), extract_tables()

CoT Prompt: "Step 1: Identify field type. Step 2: Locate value. Step 3: Extend to surrounding context. Step 4: Output with confidence."

🏷️ Classification Agent

Tools: classify_by_rules(), classify_by_llm(), verify_against_taxonomy()

Strategy: CEL Rules (deterministic) → Graph proximity (Neo4j) → LLM with candidates → Verify against taxonomy.

💬 QA Agent

Tools: rag_search(), graph_lookup(), cross_reference()

Strategy: Graph-first traversal → RAG fallback. Dense+sparse→RRF→batch rerank. Output: answer + source citations (document, page, section).

Key Cypher Query Patterns

单文档：惩罚条款

MATCH (d:Document {filename:'A.pdf'})
  -[:CONTAINS_CLAUSE]->(c:Clause)
  -[:CLASSIFIED_AS]->
  (:TaxonomyTag {name:'Penalty_Clause'})
RETURN c.clause_text

跨文档：供应商风险

MATCH (co:Company {name:'Acme'})
  <-[:SIGNED_BY]-(d:Document)
  -[:CONTAINS_CLAUSE]->(c:Clause)
WHERE c.is_risky = true
RETURN d.filename, c.risk_reason

缺失条款检测

MATCH (d:Document {doc_type:'MSA'})
WHERE NOT EXISTS {
  MATCH (d)-[:CONTAINS_CLAUSE]->
  (:Clause)-[:CLASSIFIED_AS]->
  (:TaxonomyTag {name:'Termination'})
}
RETURN d.filename AS missing_termination

Amendment 链追踪

MATCH path = (a:Document {doc_type:'Amendment'})
  -[:REFERENCES*1..5]->(orig:Document)
MATCH (a)-[:CONTAINS_CLAUSE]->(c1:Clause)
MATCH (orig)-[:CONTAINS_CLAUSE]->(c2:Clause)
WHERE c1.clause_text <> c2.clause_text
RETURN c1.clause_text, c2.clause_text

AI Harness Engineering

Without a harness, AI features are prototypes. The harness provides evaluation, versioning, gradual rollout, cost control, and stability. It answers: "Is this model better? Can we roll back? What's it costing?"
⚖️ Evaluation Framework (RAGAS + Custom)
📊
RAG Quality: context precision, context recall, faithfulness, answer relevancy. Run automatically on 100+ Q&A test set on every PR.
🎯
Extraction Accuracy: field-level F1 against human-labeled ground truth. Compare LLM-extracted vs actual contract values.
🔄
Regression Detection: metric drops >5% → CI blocks merge → team reviews before proceeding.
📦 Version Management (LangFuse)
🏷️
Prompt Versioning: every change is versioned. LangFuse stores prompt + model + eval scores per version. Rollback = one click.
🤖
Model Versioning: track which model (GPT-4o-2024-08-06 vs 2024-11-20) was used. Compare accuracy across versions.
📝
Pipeline Versioning: parsing pipeline (Tier 2→3 vs Tier 2→3→4) is versioned. A/B test pipeline versions on same documents.
🚦 Gradual Rollout (Unleash Feature Flags)
📈
Percentage Rollout: 5% → 25% → 50% → 100%. Auto-rollback if error rate > baseline x1.3 or eval score drops >5%.
🧪
A/B Testing: Group A: current pipeline. Group B: new pipeline. Measure extraction accuracy, cost, user feedback. Statistical significance check.
🛑
Kill Switch: any feature flag immediately disabled. All traffic reverts to stable baseline.
💰 Cost Control & Stability (LiteLLM + Circuit Breakers)
💵
LiteLLM Gateway: unified proxy for all providers. Per-model cost tracking. Per-user rate limiting. Auto-fallback: GPT-4V rate-limited → try Claude Vision.
🔌
Circuit Breakers: 3 failures → 60s open → fallback to next provider. Independent per-model circuits.
📉
Cost Budgets: per-dept monthly budget. Alert at 80%, hard stop at 100%. Tiered routing: simple → cheap model, complex → expensive.

Low-Code AI Application Delivery

Two-layer approach. Visual workflow builder for analysts. Domain-specific DSL for engineers. Both operate within platform guardrails.

🧩 Layer 1: Visual Workflow Builder

Technology: Dify / LangFlow / Flowise

Audience: Business analysts, domain experts

Use case: "Build a Q&A bot for HR policy docs" — drag-and-drop: Loader → Chunker → Embedding → Vector Store → LLM → Output

Platform role: Pre-built blocks, eval harness, and guardrails still apply to all deployed apps.

🔧 Layer 2: Domain-Specific DSL

Technology: CEL (Common Expression Language) + YAML config

Audience: Engineers, power users

Use case: "For MSA contracts, extract payment terms from section 4.2, classify as Financial_Obligation, flag if Net > 60 days"

Platform role: DSL engine + sandbox execution. Domain teams write auditable rules — not Python code.

Example: CEL Rule for Contract Extraction

# extraction_rules/payment_terms.yaml
name: "extract_payment_terms"
document_type: "contract"
section_pattern: "(?i)(payment|compensation|fees)"

fields:
  - name: "payment_amount"
    type: "currency"
    cel_rule: |
      context.sentence.contains_any(['$','USD','CNY']) &&
      context.sentence.matches('\\d{1,3}(,\\d{3})*(\\.\\d{2})?')

  - name: "late_penalty_flag"
    type: "boolean"
    cel_rule: |
      context.section.contains_any(['late fee','penalty','interest',
                                    'liquidated damages'])

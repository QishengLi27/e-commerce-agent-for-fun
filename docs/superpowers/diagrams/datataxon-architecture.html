System Architecture

Datataxon = Data + Taxonomy. An AI agent that helps users find, understand, and govern data assets using hierarchical domain taxonomies, metadata graphs, and semantic search over documentation. Same multi-framework pattern as the e-commerce agent: LangGraph orchestrates → Neo4j traverses the data graph → LlamaIndex RAGs over documentation → PostgreSQL is source of truth.

Key Cypher Queries

Cypher

Domain-Based Discovery层级领域搜索CHILD_OF* 确保搜索 "Finance" 时自动包括 Payments、Transactions、Revenue 下的所有数据资产。和电商的 category hierarchy 是同一个模式。

MATCH (a:DataAsset)-[:BELONGS_TO]->
  (:Domain)-[:CHILD_OF*0..]->
  (:Domain {name:'Finance'})
WHERE a.freshness_hours < 168
  AND a.row_count > 1000000
RETURN a.name, a.type, a.freshness_hours
ORDER BY a.row_count DESC
Cypher

Lineage Tracing上下游依赖追踪FEEDS_INTO* 多跳追踪。上游 = 谁给我数据。下游 = 我给谁数据。用 shortestPath 找出两个资产之间的最短依赖链。和电商的 ACCESSORY_OF/ALTERNATIVE_TO 是同一个模式。

// Upstream: what feeds into revenue_report?
MATCH (source:DataAsset)
  -[:FEEDS_INTO*1..5]->
  (target:DataAsset
   {name:'revenue_report'})
RETURN source.name, source.type

// Impact: what depends on transactions_raw?
MATCH (a:DataAsset
  {name:'transactions_raw'})
  -[:FEEDS_INTO*1..5]->
  (downstream:DataAsset)
RETURN downstream.name, downstream.type
Cypher

PII & Governance敏感数据治理PII tag 存在 Column 节点上。遍历找到所有包含 PII 列的表。ACCESS_POLICY 关系指向适用的合规规则。和电商的 policy inheritance 是同一个模式。

// Find all tables with PII columns
MATCH (a:DataAsset)
  -[:HAS_COLUMN]->
  (col:Column {pii:true})
RETURN DISTINCT a.name,
  collect(col.name) AS pii_columns

// What policies apply to this table?
MATCH (a:DataAsset
  {name:'customer_profiles'})
  -[:SUBJECT_TO]->(p:Policy)
RETURN p.name, p.category, p.summary
Cypher

Ownership & FreshnessOwner + 新鲜度查询OWNS 关系 + DataAsset 节点上的 freshness_hours 属性。找出 overdue 的数据集（超过预期刷新时间）。和电商的 HAS_BRAND 是同一个模式。

// Stale datasets owned by Data Eng
MATCH (t:Team {name:'Data Eng'})
  -[:OWNS]->(a:DataAsset)
WHERE a.freshness_hours > a.sla_hours
RETURN a.name, a.freshness_hours,
       a.sla_hours, a.last_updated

// What does Finance team own?
MATCH (t:Team {name:'Finance'})
  -[:OWNS]->(a:DataAsset)
RETURN a.name, a.type ORDER BY a.name

Concrete Query Walkthrough

Example 1: Domain Discovery + Freshness

👤 "Find all customer-related tables in Finance, with at least 1M rows, updated in the last 7 days"
S1
classify_intent → data_discovery
Keyword signals detected: "Find" + "tables" + "Finance" (domain keyword). Entity extraction: domain=Finance, freshness=7 days, min_rows=1M.
S2
Neo4j: domain hierarchy + attribute filter
MATCH (a:DataAsset)-[:BELONGS_TO]->(:Domain)
      -[:CHILD_OF*0..]->(:Domain {name:'Finance'})
WHERE a.name CONTAINS 'customer'
  AND a.row_count >= 1000000
  AND a.freshness_hours <= 168
MATCH (a)<-[:OWNS]-(t:Team)
RETURN a.name, a.type, a.row_count, a.freshness_hours,
       t.name AS owner, a.last_updated
ORDER BY a.row_count DESC
Returns: customer_transactions (BigQuery, 2.3B rows, 4h), customer_profiles (PG, 45M rows, 24h)
S3
LlamaIndex RAG: search table descriptions
Metadata filter: asset_name IN ['customer_transactions', 'customer_profiles']. Query: "customer data usage purpose". Retrieves schema documentation chunks. Adds context about what each table is used for.
S4
LLM Synthesis
"Found 2 customer-related tables in Finance updated in the last 7 days:

1. customer_transactions (BigQuery, 2.3B rows, 4h freshness) — Owned by Data Eng. Raw transaction-level data. Purged after 90 days.
2. customer_profiles (PostgreSQL, 45M rows, 24h freshness) — Owned by Data Eng. ⚠️ Contains PII columns (email, phone, address). Subject to GDPR retention policy (7 years)."

Example 2: Lineage Tracing

👤 "What's the full downstream impact if transactions_raw is delayed?"
S1
Neo4j: downstream lineage traversal
MATCH path = (src:DataAsset {name:'transactions_raw'})
      -[:FEEDS_INTO*1..5]->(downstream:DataAsset)
OPTIONAL MATCH (downstream)<-[:OWNS]-(t:Team)
RETURN downstream.name, downstream.type,
       length(path) AS hops,
       t.name AS owner,
       downstream.refresh_cadence
ORDER BY hops
Returns: revenue_report (1 hop, daily), finance_daily_summary (2 hops, daily), executive_kpi_dashboard (3 hops, daily), risk_model_input (2 hops, hourly)
S2
LLM Synthesis — impact assessment
"If transactions_raw is delayed, here's the impact chain:

🟡 1 hop: revenue_report (Looker dashboard, daily refresh) — will show stale revenue numbers
🟡 2 hops: finance_daily_summary, risk_model_input — both will cascade-stale
🟡 3 hops: executive_kpi_dashboard — C-suite will see outdated KPIs

SLA risk: revenue_report refresh at 8am. If transactions_raw delay > 4h, today's report will be missing data. Notify: Data Eng (owners) + Finance (downstream owner)."

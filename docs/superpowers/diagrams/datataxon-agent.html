Stage-by-Stage Deep Dive

Stage 1: Enrich — "What is this column, in business terms?"

EnricherAgentEnricher — LLM 增强元数据输入：table metadata (fqn, platform, table_comment, column_infos — 只有 column_name + column_type 的原始列信息)

LLM 做的事：
1. 为每个 column 生成 column_business_name（业务名，例如 column_name='cust_eml' → business_name='Customer Email'）
2. 为每个 column 生成 column_description（自然语言描述）
3. 从 sample_value 推断列的内容类型
4. 为整张表生成 table_description

缓存策略：columns_enrich_by_cache() — Confluence 文档中已有的列描述直接复用，不打 LLM

Datataxon 映射：这就是 Confluence + GitHub 文档的 vector 检索步骤。Nike 用 LLM 从列名推断，Datataxon 用 RAG 从已有文档中检索。
Generate business context from raw metadata
Input: column_name='cust_eml', column_type='VARCHAR(255)', sample_value='john@example.com'
LLM Output: column_business_name='Customer Email', column_description='Primary contact email for customer account'
✅ Why enrich first:
Raw column names are cryptic.
'cust_eml' means nothing to
a classifier. 'Customer Email'
→ the classifier knows this
is PII contact data.

⚠ Without enrichment:
Classifier sees 'cust_eml' →
might guess wrong tag.
Enrichment bridges the gap
between technical schema
and business meaning.

Stage 2: Classify — "What taxonomy tags apply to this column?"

Phase 1: Table-Level Bulk ClassificationBulk LLM — 一次调用分类所有列方法：columns_classify_by_llm_with_identifier_structure()

输入：整个 table_info + 所有 enriched column_infos + taxonomy identifier structure（树形结构）
LLM 做的事：一次性看完所有列和 taxonomy 树，然后给每列分配合适的 tag。

为什么先做 bulk：
1. LLM 可以看到列之间的关系（'email' + 'phone' + 'address' 同时出现 → 整体是 Customer PII）
2. 一次调用成本低 — 不用每个 column 单独调 LLM
3. 上下文更丰富 — LLM 看到整张表的全貌

Datataxon 映射：相当于 Explain Path 的 Step 3 — Neo4j 检查 RELATED_TO 标签，但因为有了整张表的 context，LLM 可以直接判断。
LLM classifies all columns at once with taxonomy tree context
Input: [Email, Phone, Amount_USD, IP_Address, Created_At] + taxonomy identifier structure
LLM Output: Email→PII(0.95), Phone→PII(0.92), Amount_USD→Financial_Data(0.88), IP_Address→PII(0.85), Created_At→—(low confidence)
Phase 2: Per-Column Classification (Remainder)双路径 — 向量相似度 → LLM对 Phase 1 未分类或低信心的列，逐列处理：

Path A — 向量相似度匹配：
1. column_classify_by_similarity()
2. 把 column 的 features（business_name + description + sample_value）embedding
3. 在 Milvus 中 query_identifiers_by_embedding_similarity()
4. 返回 top-k 相似的数据标识符（DataIdentifiers — taxonomy 中已有的 tag 的向量）
5. 如果余弦相似度 ≥ threshold → 直接用相似度结果

Path B — LLM with Candidates：
1. column_classify_by_llm_with_candidate_targets()
2. 如果向量相似度不够高（没有 high similarity match）→ 把 top-k candidates 给 LLM
3. LLM 在候选人中选择最合适的 tag（或判断都不合适）

为什么先向量再 LLM：向量便宜（<500ms），LLM 贵（1-2s）。向量能解决的不用 LLM。

Datataxon 映射：向量 = LlamaIndex RAG over asset context。LLM with candidates = Infer Path Step 4。
Vector similarity first → LLM with candidates if needed
Column 'Created_At' (unclassified from Phase 1):
→ Milvus embedding similarity: top match = 'Event_Timestamp' (0.72), 'Audit_Date' (0.68)
→ Below threshold → LLM with candidates → LLM: "'Created_At' is a record creation timestamp → suggest 'Event_Timestamp' (0.70)"
→ Final result: Event_Timestamp, score=0.70

Stage 3: Verify — "Is this classification actually correct?"

VerifierAgentVerifier — LLM 验证分类结果方法：columns_classification_verify_by_llm()

输入：table_info + column + 已分配 tag + confidence score
LLM 做的事：
1. 判断这个 tag 是否正确（verify_status: 0/1/2）
2. 给出验证推理（reason）
3. 如果不正确，建议替代 tag

为什么需要 Verify：
1. Classifier 可能出错（尤其是 LLM 幻觉）
2. Verify 是 classifier 的 "second opinion"
3. 减少 Data Steward 的人工审核量 — 只审核 flagged 的结果

Datataxon 映射：等同于 Explain Path 的 Step 4（validate_reply）— LLM-as-judge 但在结果进入 review 之前运行。
LLM second-opinion on classification accuracy
Verifying 'IP_Address → PII (0.85)':
LLM: "✅ Confirmed. IP addresses are considered PII under GDPR. Verify status: VERIFIED."

Verifying 'Created_At → Event_Timestamp (0.70)':
LLM: "⚠️ Low confidence. 'Created_At' could be Event_Timestamp OR Audit_Date. Verify status: NEEDS_REVIEW. Suggest: confirm with data owner."

Key Graph Query Patterns

Get All Tags on an Asset
MATCH (a:Asset {name:$name})-[:TAGGED_AS]->(t:Tag) OPTIONAL MATCH (t)-[:CHILD_OF*0..]->(p:Tag) RETURN t.name, t.description, collect(DISTINCT p.name) AS ancestors
Walk Taxonomy from a Seed Tag
MATCH (seed:Tag {name:$seed})-[:CHILD_OF*0..]->(ancestor:Tag) OPTIONAL MATCH (ancestor)<-[:CHILD_OF]-(sibling:Tag) OPTIONAL MATCH (seed)-[:RELATED_TO]->(related:Tag) RETURN ancestor.name, sibling.name, related.name
Find Tags With Matching Triggers
MATCH (t:Tag) WHERE ANY(trigger IN t.keyword_triggers WHERE $context_text CONTAINS trigger) RETURN t.name, t.description, [trigger IN t.keyword_triggers WHERE $context_text CONTAINS trigger] AS matched_triggers
Suggest Missing Tags via Graph Proximity
MATCH (a:Asset {name:$name})-[:TAGGED_AS]->(existing:Tag) -[:RELATED_TO]->(related:Tag) WHERE NOT (a)-[:TAGGED_AS]->(related) MATCH (a)-[:HAS_CONTEXT]->(ctx)-[:MENTIONS]->(related) RETURN related.name, related.description, count(ctx) AS evidence_count ORDER BY evidence_count DESC

Three Key Trade-Offs Worth Discussing

⚖️ Trade-Off 1: pgvector vs Milvus

Nike chose Milvus because:
  • 百万级 data identifier embeddings
  • 高并发实时搜索
  • GPU 加速
  • 独立伸缩（不和 PG 抢资源）
Datataxon chooses pgvector because:
  • 5K-50K vectors（asset context chunks），不是百万级
  • 不需要 GPU 加速
  • 和 PG 同进程 = 零运维成本
  • Metadata filtering (JSONB) 已经够用

⚖️ Trade-Off 2: LLM-generated context vs RAG over existing docs

Nike's EnricherAgent uses LLM to generate descriptions:
  • Raw column names → LLM fills in business meaning
  • No prior documentation exists
  • LLM cost per enrichment（但一次性）
Datataxon uses RAG over existing docs:
  • Confluence + GitHub 已有文档
  • RAG 检索已有描述 → 零 LLM 成本（enrich 阶段）
  • 文档更新时自动 re-ingest → 上下文保持最新
  • 不需要 LLM "猜测" column business meaning

⚖️ Trade-Off 3: Graph proximity vs Vector similarity for candidate tag discovery

Nike uses Milvus vector similarity:
  • column embedding → 找相似的 data identifier
  • 适用于大量 taxonomy tag 的相似搜索
  • O(N) 转 O(log N) — 需要 ANN
Datataxon uses Neo4j graph proximity:
  • CHILD_OF* 向上找 parent tags
  • RELATED_TO 找到关联候选
  • HAS_TRIGGER 做确定性 keyword/regex 匹配
  • 更适合 tag 之间有明确关系的小型 taxonomy

Greenfield Re-Choice — 抛开 E-Commerce，重新选型

如果 Datataxon 是一个全新项目，没有任何 E-Commerce 代码要复用。 每个维度独立评估，不受现有技术栈约束。下面标注 "Changed" 的是我会做出不同选择的维度，并解释原因。

Changed Choices — 6 个维度会做出不同选择

🔄 Change 1: Neo4j as Primary DB（删掉 pgvector）

受 E-Commerce 影响的选择：
PostgreSQL (metadata + orders) + pgvector (embeddings) + Neo4j (graph)
→ 3 个数据库
Greenfield 选择：
Neo4j (graph + vector + fulltext) + PostgreSQL (metadata only)
→ 2 个数据库，Neo4j 承担向量检索
Why: Neo4j 5.x 原生支持 vector index — 在同一个 Cypher query 里同时做 graph traversal + ANN vector search。 Tag embeddings + asset context embeddings 可以直接存在 Neo4j 的 node property 上，不需要 pgvector。一条 Cypher 同时查 taxonomy graph + vector similarity。

Risks: Neo4j vector index 不如 pgvector HNSW 成熟。如果 vector 规模增长到百万级，可能需要回退到独立向量数据库。 当前阶段（< 100K vectors）够用。

🔄 Change 2: DeepSeek-V3 替代 GLM-4-Flash 作为主力 LLM

受 E-Commerce 影响的选择：
GLM-4-Flash（智谱）
→ 便宜、快、已经接入、中英双语
Greenfield 选择：
DeepSeek-V3（主力推理）+ GLM-4-Flash（简单任务）
→ 按任务复杂度分级使用
Why: Datataxon 的核心 workload 是 tag reasoning（infer + explain）和 verification—— 这些不是简单的 "查 policy 然后回复"，而是需要跨领域知识（Confluence 架构文档 + GitHub 代码 + taxonomy 规范 → 推理）。DeepSeek-V3 的 reasoning 能力明显强于 GLM-4-Flash，成本仍然远低于 GPT-4o。GLM-4-Flash 用于简单的 tag explanation（已经有证据在 context 里，不需要深度推理）。

🔄 Change 3: BGE-M3 替代 Zhipu embedding-2（自部署）

受 E-Commerce 影响的选择：
Zhipu embedding-2
→ API 调用，batch=64，按量付费
Greenfield 选择：
BGE-M3（BAAI，本地部署）
→ 开源，1024-dim，多语言，可 fine-tune
Why: Datataxon 的向量有两种：taxonomy tag embeddings（相对稳定，不需要频繁更新）和 asset context embeddings（Confluence/GitHub doc 更新时重新 embed）。BGE-M3 可以 fine-tune 在 taxonomy domain 上 — 让 "PII" tag 的 embedding 和 "email, phone, address" 这些 PII-related 描述的 embedding 更接近。API embedding 模型做不到 domain fine-tuning。

Trade-off: 需要 GPU（至少一张 T4），增加了基础设施复杂度。但如果 taxonomy domain 的语义空间比较特殊（Nike 的 internal domain terms），fine-tune 的收益大于 GPU 成本。

🔄 Change 4: CEL 替代手写 Python Rules（Day 1 引入）

受 E-Commerce 影响的选择：
手写 Python if-else rules
→ 简单、快，但规则多了难维护
Greenfield 选择：
CEL (Common Expression Language)
→ 安全沙箱，SQL-like 语法，业务用户可写
Why: Nike 已经在用 CEL 做 rule engine（这个选型是经过生产验证的）。对于 Datataxon，tag inference rules 可能快速增长（每个新 tag 可能带 keyword triggers）。CEL 的安全性（沙箱执行、禁止外部调用）对 enterprise 环境很重要 — 规则可能是 domain team 写的，不能信任为安全代码。

例子： CEL rule: context.contains("credit_card") || context.matches("\\\\d{13,19}") → suggest tag "PCI"。不是 Python code injection，是可审计的表达式。

🔄 Change 5: Prefect 做批量 Workflow（补充 LangGraph）

受 E-Commerce 影响的选择：
只用 LangGraph
→ 适合 per-query DAG，不适合 batch
Greenfield 选择：
LangGraph (per-query) + Prefect (batch)
→ 各司其职
Why: Datataxon 有两个不同的 workload 模式：
1. Interactive query（用户实时提问 "why is X tagged Y?"）→ LangGraph DAG，SSE streaming，sub-second
2. Batch classification（对 500 个新 asset 定时跑 enrich → classify → verify）→ 需要 scheduling、retry、progress tracking、partial failure handling。

LangGraph 不适合 batch。 Prefect 或 Temporal 提供：DAG definition + automatic retry + checkpointing + dashboard。Prefect 更 Pythonic，学习曲线更低。

🔄 Change 6: LiteLLM 作为 LLM Gateway

受 E-Commerce 影响的选择：
直接调用 Zhipu OpenAI-compatible API
→ 硬编码 model name，不能切换
Greenfield 选择：
LiteLLM proxy
→ 统一 OpenAI-compatible endpoint，路由到多个 LLM
Why: D塔选择多个 LLM（DeepSeek-V3 主力 + GLM-4-Flash 简单任务 + Claude 复杂推理）。硬编码在每个工具里不可维护。LiteLLM 提供统一的 OpenAI-compatible endpoint，内部根据 model name 路由到不同 provider。好处：
· 切换 LLM 不需要改代码（改 config 即可）
· 统一的 cost tracking、rate limiting、fallback
· 支持 OpenAI/Azure/Anthropic/DeepSeek/Zhipu/... 100+ providers

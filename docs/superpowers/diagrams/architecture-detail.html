4-Stage Modular RAG Pipeline

这是 retrieval.py 中的手写 RAG——每一层独立调优，不是框架默认配置Modular RAG业界趋势（Cohere、Anthropic 推荐）：Ingestion、Retrieval、Re-rank、Resilience 各自独立选型，不绑定单一框架。每一层选择该层最好的组件。。

🔢

Dense Retrieval稠密检索 = 语义搜索原理：把用户 query 转成 1024 维向量（embedding），在 pgvector 中做 ANN（近似最近邻）搜索。

vector(1024)：句子被压缩成 1024 个数字。相似的句子 → 向量距离近。

HNSW：层次化可导航小世界图。把 200 个向量组织成多层图结构，查最近邻不需要遍历全部 200 个 — 跳表式查找，2-5ms 返回 top-5。

Cosine Distance：衡量向量夹角的余弦值。越小 = 越相似。不同于欧氏距离（几何距离），余弦距离更关注方向而非绝对位置。

擅长："退货" 匹配 "return policy"（语义相似但用词不同）
弱点："headphones" 和 "earbuds" 的 embedding 很接近，但它们的 return policy 不同 — 向量分不开不同品类的政策

Sparse Retrieval稀疏检索 = 关键词匹配原理：把 query 拆成词，用 BM25 算法给每个 chunk 打分。

BM25（Okapi Best Match 25）：TF × IDF 的升级版。TF (Term Frequency) — 词出现次数；IDF (Inverse Document Frequency) — 词有多稀有。例如 "return" 在许多文档中不出现 → 高 IDF → 出现时权重高。

k1=1.5, b=0.75：TF 饱和参数和长度归一化参数。k1 控制 TF 的增长速度（1.5 = 出现 2 次 vs 1 次的差距比线性大）；b 控制文档长度的影响（0.75 = 长文档有较弱的归一化惩罚）。

Pickle 缓存：索引序列化到 data/bm25_index.pkl，重启不需要重建。

擅长：精确关键词（"headphones" → 包含 "headphones" 的 chunk）
弱点：不同表达方式（"return" vs "send back"）— 不同词，BM25 无法匹配

RRF Fusion倒数排序融合 = 合并两个排名公式：score(doc) = Σ 1/(k + rank)

k=60：控制融合平滑度的参数。k 越小 → 高排位 doc 的优势越大；k 越大 → 更平均。60 是经典值，在大多数数据集上效果最好。

Dense 排 #1，Sparse 排 #3：1/(60+1) + 1/(60+3) = 0.0164 + 0.0159 = 0.0323
另一个 doc 两个列表都排 #2：1/(60+2) + 1/(60+2) = 0.0161 + 0.0161 = 0.0322

共识 boost：在两个列表都出现的 doc 获得更高分。
互补保留：只在一个列表高排名的 doc 不会消失，只是得分略低。

为什么不用加权求和？RRF 不需要调权重 — 两个列表的单位不同（cosine distance vs BM25 score），直接加权需要大量实验确定 α 和 β。RRF 用排名而不是原始分数，天然跨域可比。

零延迟：纯数学运算，没有 API 调用。

Σ 1/(k+rank)

Batch LLM Re-rank批量 LLM 精排 = 质量门禁一次 LLM 调用评所有 5 个 doc：不是逐文档调用（5 doc = 5 次 LLM = 2.5-10s），而是把所有 doc 放进一个 prompt，一次调用评所有。

Prompt 格式："[1] {doc1}\n[2] {doc2}...\nScores (format: [N] score):"
解析：regex 提取 [N] score 格式，clamp 到 [1, 10]
过滤：保留 score ≥ 7 的 doc。如果全部 < 7，返回最高分的 1 个（safety net）

GLM-4-Flash, temperature=0：用最快的模型 + 零温度保证评分一致。

5x 快于框架默认：LlamaIndex 默认 LLMRerank 是逐文档调用。这个 batch 优化把延迟从 2.5-10s 降到 0.5-2s。

为什么需要这一层？RRF 融合后 top-5 可能包含 2-3 个不相关的 doc。直接喂给 LLM 会降低回答质量。Re-rank 把不相关的排到后面（或直接过滤掉）。

Neo4j — Graph Database for Structured Reasoning

Neo4j为什么需要图数据库SQL 能 JOIN，但 JOIN 不是 graph traversal。

"Find smartphones under $800, not Apple, with 256GB storage, AND their accessories from different brands, AND the policies that apply to those accessories" — 用 SQL 写需要 4 个 JOIN + 2 个子查询 + recursive CTE。用 Cypher 是一个自然的多跳路径表达式。

Neo4j Community Edition：免费，单节点，足够的 scale。Neo4j 在这个项目中的角色是 read-only graph cache — PostgreSQL 是 source of truth，Neo4j 从 PG 同步数据，查询时只读。 是纯图数据库，用 Cypher 语言做多跳关系查询。和 SQL JOIN 的本质区别：Cypher 把关系表达为图中的边（edge），遍历是 O(1) 的指针追踪，不是 O(N) 的扫描。

Key Cypher Query Patterns

Cypher

Synonym-Aware Product LookupFulltext + SynonymNeo4j 的 Lucene fulltext index 索引 Product 节点的 name + search_terms（synonyms）。

"苹果手机" → fulltext 搜索 → "iPhone 15 Pro"
"索尼耳机" → fulltext 搜索 → "Sony WH-1000XM5"

Synonyms 在 neo4j_setup.py 中从 entity_synonyms 表读取，写入 Product.search_terms 数组属性。

CALL db.index.fulltext.queryNodes(
  'product_search', '苹果手机')
YIELD node, score
WHERE score > 0.5
RETURN node.name, node.price, score
ORDER BY score DESC LIMIT 1
Cypher

Category Path & Policy InheritanceCHILD_OF* 多跳遍历[:CHILD_OF*0..4] 表示 0 到 4 跳。0 跳 = 自身。所以查询会找到 iPhone 所在的所有 ancestor category。

Policies 挂在 Electronics 上。通过 CHILD_OF*，Flagship Phones → Smartphones → Mobile Devices → Electronics → 找到 policies。

DISTINCT 去重：同一个 policy 可能通过不同 ancestor 被多次找到。

MATCH (p:Product {name:'iPhone 15 Pro'})
  -[:IN_CATEGORY]->(:Category)
  -[:CHILD_OF*0..4]->(anc:Category)
  -[:HAS_POLICY]->(pol:Policy)
RETURN DISTINCT pol.name, pol.summary,
       pol.policy_type
Cypher

Attribute-Filtered Product Search多约束候选搜索WHERE 子句做属性过滤。
EXISTS 子查询检查属性值（EAV 模型的图版本）。
NOT (p)-[:HAS_BRAND]->(:Brand {name:'Apple'}) 排除 Apple 产品。

Neo4j 的 Cypher planner 自动优化执行顺序 — 先用 fulltext index 找到候选产品，再用属性过滤。

MATCH (p:Product)
  -[:IN_CATEGORY]->(:Category)
  -[:CHILD_OF*0..]->(:Category
    {name:'Smartphones'})
WHERE p.price < 800
  AND NOT (p)-[:HAS_BRAND]->
    (:Brand {name:'Apple'})
RETURN p.name, p.price
ORDER BY p.price DESC
Cypher

Product Relations Traversal关系双向查询ACCESSORY_OF 是有向关系。但查询时用双向匹配（← 和 → 都查）确保覆盖。

UNION 合并两个方向的结果。
OPTIONAL MATCH 获取 accessory 的 category 名，不强制要求有 category。

MATCH (p:Product
  {name:'iPhone 15 Pro'})
  -[:ACCESSORY_OF]->(a:Product)
RETURN a.name, a.price
UNION
MATCH (a:Product)
  -[:ACCESSORY_OF]->(p:Product
    {name:'iPhone 15 Pro'})
RETURN a.name, a.price

LlamaIndex — RAG Framework for Ingestion & Querying

LlamaIndexLlamaIndex 在项目中的角色不是替代 hand-built RAG。LlamaIndex 负责两件事：

1. Ingestion（数据预处理）：SentenceWindowNodeParser 做智能 chunking，替换现有的 RecursiveCharacterTextSplitter(500,50)。这是 hand-built RAG 最弱的一层。

2. Query Engine（查询引擎）：可选。对于需要 metadata-filtered vector search 或 query decomposition 的场景，用 LlamaIndex QueryEngine。简单场景继续用 hand-built RAG。

Hand-built RAG 保留：dense+sparse→RRF→batch rerank+circuit breaker 全部保留 — 这些是 hand-built RAG 比 LlamaIndex 默认做得更好的部分。 在项目中的角色是 Ingestion 层 + 可选 Query Engine。不是替代 hand-built RAG——它是增强 hand-built RAG 最弱的一层（chunking），同时保留最强的部分（retrieval + re-rank + resilience）。

Ingestion Pipeline声明式数据预处理管道LlamaIndex 的 IngestionPipeline 是声明式的 — 你指定 transformations（chunking → embedding → store），框架处理执行。

对比 hand-built：需要手动调用 TextLoader、TextSplitter、Embedding、PGVector.from_documents()。LlamaIndex 把这些封装成一个 pipeline 对象。 — 从 product_descriptions.txt 到 pgvector

Guardrails — 多层防护体系

Guard 的核心原则：不依赖 LLM 做 Guard为什么 Guard 不应该依赖 LLMLLM 可能出错、可能被 prompt injection 绕过、可能对同一个 query 给出不同的判断。最可靠的 guard 是正则匹配和字符串校验 — 它们不需要"判断"，只需要"检查"。Guard 本身如果有不确定性，就失去了 guard 的意义。，因为 LLM 也会出错。最可靠的 Guard 是正则匹配和确定性校验。

Resilience: 故障时的降级路径

Circuit Breaker断路器模式状态机：Closed（正常） → Open（熔断） → Half-Open（探测恢复）
参数：failure_threshold=3, recovery_timeout=60s
行为：连续 3 次失败 → 熔断 60 秒。60 秒后允许 1 次探测调用 → 成功则恢复（Closed），失败则继续熔断（Open）。

为什么 3 次而不是 5 次或 1 次？1 次会因偶发网络抖动而误熔断。5 次会让用户在故障后等待太久。
为什么 60 秒？LLM API 的恢复通常在 30-120 秒内。60 秒是经验值。

两个独立的 CB：embedding_circuit（pgvector 搜索）和 llm_circuit（re-rank）。一个失败不影响另一个。

Full Glossary

Every technical term used in this project, with plain-language explanations.

Agent

LangGraph StateGraph

一个有向无环图（DAG）来表示 agent 的执行流程。每个节点是纯函数 State → State。与 ReAct agent 的区别：执行路径在构建时确定，不是 LLM 运行时决定。这意味着可以精确推理 agent 的行为 — 不会出现"LLM 决定跳过 tool 调用"的意外。

Agent

ReAct Agent

Reasoning + Acting 的缩写。LLM 在每个步骤生成一个"思考 → 行动"对。自主决定调用哪个 tool、什么时候停止。灵活但不稳定 — 可能循环、跳过必要的 tool call、或 hallucinate tool 调用。本项目的 LangGraph DAG 是对 ReAct 的替代方案。

Agent

Intent Classification

把用户自然语言归类到已知的 6 个意图之一。三档策略：keyword（regex 规则引擎）、llm_hybrid（LLM 抽取意图+实体）、semantic（embedding 搜索候选 + LLM 在候选内分类，应对 10K+ SKU 的场景）。

RAG

Dense Retrieval

用 embedding 模型把 query 转成 1024 维向量，在向量数据库中做 ANN（近似最近邻）搜索。衡量语义相似度 — "退货"能匹配"return policy"即使没有共同的词。弱点：同品类不同政策（headphones vs earbuds）的 embedding 太接近，检索不准确。

RAG

Sparse Retrieval (BM25)

BM25（Okapi Best Match 25）是 TF-IDF 的升级版。把 query 拆成词，用词频×逆文档频率给每个文档打分。擅长精确关键词匹配 — "headphones"一定匹配包含"headphones"的 chunk。弱点：同义词（"return" vs "send back"）不匹配。

RAG

RRF (Reciprocal Rank Fusion)

公式 score(doc) = Σ 1/(k + rank)。合并两个排序列表（dense + sparse），不需要人工权重。参数 k=60 控制融合平滑度。文档在两个列表中都出现时获得共识 boost，只在一个列表出现也不会消失。

RAG

Batch LLM Re-rank

一次 LLM 调用评所有检索文档的质量（1-10 分）。与逐文档评分的区别：5 个文档 = 1 次调用 vs 5 次调用，延迟和成本降低 5 倍。保留分数 ≥7 的文档。Temperature=0 保证一致性。

Vector

Embedding (向量嵌入)

把一段文本压缩成固定长度的数字数组（这里用 1024 维）。语义相似的文本 → 向量距离近。Zhipu embedding-2 支持中英双语，batch 大小 64。

Vector

pgvector + HNSW

pgvector 是 PostgreSQL 的向量扩展。HNSW（Hierarchical Navigable Small World）是向量索引算法 — 把向量组织成多层图结构，查询不需要遍历所有向量（O(log N) 而非 O(N)）。200 个向量 2-5ms 返回 top-5。

Vector

Cosine Distance

衡量两个向量夹角的余弦值。0 = 完全相同方向，1 = 正交（无关）。比欧氏距离更适合语义搜索 — 关注方向而非绝对位置。两个相似句子的向量方向一致，即使长度不同。

Graph

Cypher (图查询语言)

Neo4j 的查询语言，相当于图数据库的 SQL。MATCH (p:Product)-[:IN_CATEGORY]->(c:Category) 表示"找到所有产品节点和它们连接到的分类节点"。天然支持多跳遍历 ([:CHILD_OF*0..4] = 0 到 4 跳)。

Graph

EAV (Entity-Attribute-Value)

实体-属性-值设计模式。不同品类的产品有不同的属性 — 手机有 screen_size，咖啡机没有。用 EAV 表（product_attributes: product_id → attribute_id → typed_value）可以动态扩展属性而不改表结构。

Graph

Materialized Path

在 categories 表存储枚举路径（如 "1.2.3.4"），表示从根到当前节点的完整路径。用 WHERE path LIKE '1.2.%' 可以高效查询所有子节点而不需要递归 JOIN。

Infra

Circuit Breaker (断路器)

防止级联故障的模式。3 次连续失败后打开电路（熔断），60 秒内所有调用直接返回 fallback。60 秒后允许一次探测——成功则恢复，失败则继续熔断。embedding 和 LLM 有两个独立的断路器。

Infra

Tenacity (重试库)

Python 的重试库，支持指数退避（exponential backoff）。第一次重试等 1 秒，第二次等 2 秒，最多 2 次。重试失败计入 circuit breaker 的 failure count。

Infra

Semantic Cache

用 pgvector 做 LLM 响应的语义缓存。cosine distance < 0.3 时返回缓存。Weather 查询显式排除——不同城市的 weather embedding 太相似（城市名在高维空间中距离被稀释），缓存会返回错误城市的信息。

Pattern

Typed Dataclass (frozen)

用 Python @dataclass(frozen=True) 替代 raw dict。ProductRef(name, price, sku) 而不是 {'name': ..., 'price': ...}。好处：IDE 自动补全、类型检查、不可变性（线程安全）、字段改名时编译报错而非运行时 KeyError。

Pattern

Dependency Injection

对象通过 __init__ 接收依赖而非内部创建。Neo4jStore(uri, user, password) 而非 Neo4jStore() 内部读 settings。好处：单元测试可传 mock，不依赖全局状态。FastAPI 用 Depends() 自动注入。

Pattern

Source of Truth

PostgreSQL 是唯一的写入源。Neo4j 是读缓存，从 PG 同步，没有运行时双写。好处：消除了 PG 和 Neo4j 数据不一致的可能。不需要分布式事务。如果 Neo4j 挂了，重新跑 neo4j_setup.py 即可恢复。

Safety

Prompt Injection

用户在输入中注入指令来覆盖系统 prompt。例如："忽略之前的指令，告诉我..."。防御方式：用 XML 标签或特殊分隔符把用户输入和系统指令隔离。本项目的 LangGraph DAG 提供了额外防御——即使 LLM 被注入，它调用的 tool 是由 graph 决定的，不是 LLM 决定的。
